סגור—להלן “הנחיית־על” אחת, תכל’ס וללא קוד, שמדריכה את הבוט איך לעבוד עם Tailwind 4.1 (הכי עדכני), לעדכן הכל להכי חדש, לנקות ישנים בלי לשבור, ולזהות “טפיליות” שמפריעות. שמרתי את זה בשפה של משימות, קריטריונים, ובדיקות—ככה הוא לא יסטה מילימטר.

⸻

0) עקרון־על
	•	המטרה: UI שנראה מוצר אמיתי, עובד חלק במובייל/דסקטופ, עם עיצוב עקבי ואפס כפילויות בתצורה/חבילות.
	•	הכלל: עדכן–אחד–נקה. לא מוסיפים שכבות חדשות לפני שמוודאים שלא נשארו דורות קודמים (Tailwind v3, PostCSS ישן, CSS מותאם שחוסם, תבניות Flask ליד React).

⸻

1) הסטאק והגרסאות שצריך להשתמש בהן
	•	React 19 + Vite 7 (או bundler מודרני שקול).
	•	Tailwind 4.1 עם המודל החדש (“CSS-first”): טעינה דרך @import "tailwindcss" בלבד, וטוקנים דרך @theme בקובץ ה־CSS הראשי.
	•	PostCSS דרך התוסף הרשמי של Tailwind (אחד בלבד, ללא פלאגינים היסטוריים).
	•	אנימציות: Motion (Framer Motion) לשכבות עדינות (fade, slide, stagger).
	•	רכיבים נגישים: Radix Primitives / shadcn (להעתקה פנימה ולהתאמה), רק כשבאמת צריך דיאלוג/Sheet/Toast.
	•	פונט: Heebo (עברית+לטינית), טעינה נכונה (preconnect), RTL גלובלי, dir="ltr" לשדות אימייל.
	•	דפדפנים יעד: Chrome מודרני, Safari 16.4+, Firefox מודרני. אם חייבים ישנים—אל תעלה ל־v4.

⸻

2) תהליך עדכון וניקוי (ללא קוד, שלבים ברורים)
	1.	סניף עדכון נפרד: פתח ענף “upgrade-ui”.
	2.	עדכון חבילות ללייטסט: השתמש בכלי עדכון תלויות סטנדרטי (כמו ncu) להעלות ל־React 19, Vite 7, Tailwind 4.1.
	3.	שדרוג Tailwind: הרץ את כלי השדרוג הרשמי של Tailwind (הוא יארגן PostCSS/CLI/תוספים ויתריע על breaking changes).
	4.	החלפת תצורה:
	•	ודא שיש קובץ CSS כניסה יחיד שמייבא Tailwind בצורה החדשה (לא פזור בקומפוננטות).
	•	העבר טוקנים (צבעים/רדיוסים/ריווחים) ל־@theme ב־CSS, במקום קובץ config ישן.
	5.	ניקוי הישן:
	•	הסר שימוש ב־@tailwind base/components/utilities.
	•	הסר פלאגינים ישנים של PostCSS שנכללים כבר ב־v4 (כגון autoprefixer/import היסטוריים).
	•	מחק חבילות CSS Framework אחרות (Bootstrap/Material/MUI) אם קיימות—אל תערבב עם Tailwind.
	•	ודא שאין Sass/Less ישן; Tailwind v4 הוא ה־preprocessor.
	6.	יישור קלאסים: אם שונו שמות/סקיילים (כמו shadow/ring/outline), עדכן את הכיתובים הוויזואליים המושפעים.
	7.	נעילת גרסאות: ודא שכל החבילות ננעלות בקובץ הנעילה; אל תיתן לגרסאות לזוז אחרי PR זה.

⸻

3) בדיקת “טפיליות” וקונפליקטים שמחרבים ריצה

בצע את הבדיקות האלה לפני ואחרי ההרצה, ותתקן אם משהו נמצא:
	•	Tailwind כפול: ודא שיש התקנה אחת של tailwindcss ושימוש אחד בלבד ב־Tailwind בקוד ה־CSS הראשי.
	•	PostCSS כפול: ודא שיש תצורה אחת—זו של Tailwind v4—ולרוץ בלי פעמיים pipeline.
	•	ייבוא כפול: ודא ש־Tailwind לא נטען גם בקבצי קומפוננטות וגם בכניסה הראשית.
	•	סטטיק כפול בצד השרת: /auth מוגש מ־React בלבד; אין תבנית Jinja שמשרתת אותו במקביל. מחק/נטרל תבניות התחברות ישנות.
	•	CSS מותאם “שחור”: אין style ישן או CSS מותאם שמבטל utilities של Tailwind. אם צריך overrides—לעולם רק כ־layer קטן ומבוקר.
	•	ספריות UI כבדות: אל תטעין גם ספריית קומפוננטים שלמה בנוסף ל־Tailwind + Radix; זה מנפח ויוצר התנגשויות.
	•	אייקונים: השתמש ב־SVGים נקודתיים (למשל Lucide React או קבצי SVG מקומיים)—לא ספריות איקונים כבדות.

⸻

4) סטנדרט עיצוב אחיד (כדי שזה ייראה מוצר)
	•	רקע חי: שני blobs גרדיאנטיים OKLCH מטושטשים בעדינות; שקיפות עדינה מעליהם.
	•	Glass Card: כרטיס שקוף מעט עם blur עדין, ring דק, radius גדול, וצל רך.
	•	טיפוגרפיה: Heebo 400/600/700; פונטים נעימים בעומס טקסט.
	•	Micro-interactions: כניסה רכה למסך (fade 150–250ms), stagger קל לשדות, hover עדין ל־CTA.
	•	נגישות: labels לכל שדה, focus ring נראה, aria-live לטוסטים, קונטרסט AA, מקשי Tab/Enter.
	•	מובייל־פרסט: טורים מתחלפים (מובייל טור אחד; דסקטופ שני טורים), tap targets ≥ 44px, בלי גלילה אופקית.

⸻

5) איך לעבוד ביום־יום עם Tailwind 4.1
	•	Atomic First: להישען על utilities של Tailwind. לא להוסיף CSS ידני אלא אם חייבים.
	•	Tokens לפני הכל: צבע/רדיוס/spacing—להגדיר פעם אחת כטוקנים ולהשתמש חוצה מסכים. זה מבטיח עקביות.
	•	Container Queries: אם רכיב צריך לשנות layout בתוך כרטיס צר—להשתמש ב־container queries של v4 במקום media queries חיצוניות.
	•	אנימציות עם Motion: כל אנימציה—שקופה, קצרה, עדינה. לא רועש.
	•	רכיבים נגישים עם Radix/shadcn: כשצריך modal/toast/sheet/dropdown—לייבא תבנית, להתאים לצבעי המותג ולטוקנים, ולהשאיר את ההיגיון נגיש.

⸻

6) “התחברות” – קריטריונים ל־GO (ללא קוד)
	•	נראות: רקע חי, Glass Card, Heebo, אייקונים בשדות, כפתור גרדיאנט.
	•	התנהגות: ולידציה inline; מצב “מתחבר…”, טוסט שגיאה/הצלחה; “שכחתי סיסמה” ו“איפוס” זורמים.
	•	ניווט: אחרי התחברות—הפניה מלאה ל־/app/admin או /app/biz לפי תפקיד; לא SPA משולב בתוך /auth.
	•	מובייל: 320–400px בלי גלילה אופקית; כפתורים ושדות לא נחתכים ע”י המקלדת; tap targets ≥ 44px.
	•	נגישות: labels, focus-ring, aria-live להודעות.
	•	Lighthouse למסך זה: 90+ ב־Performance/Accessibility/Best Practices.

⸻

7) בדיקות מהירות (לפני Merge לכל מסך חדש)
	•	רספונסיביות: 320/375/768/1024/1280—ללא גלילה אופקית, מרווחים נכונים.
	•	נגישות: Tab לוגי, Enter שולח, הודעות נקראות, קונטרסט AA.
	•	ביצועים: אין ספריות איקונים כבדות; אין תמונות ענק; CSS מינימלי.
	•	אחידות: אותו רדיוס/צל/טיפוגרפיה/צבעים כמו במסכי auth.
	•	תאימות דפדפנים יעד: Chrome/Safari/Firefox מודרניים.
	•	לוג־קונסול: נקי מאזהרות/שגיאות.

⸻

8) “ניקיון עומק” אחת לשבוע (שומר על פרויקט בריא)
	•	סריקת תלויות: לבדוק שאין שתי גרסאות של אותה חבילה (React, Tailwind, Motion).
	•	בדיקת Lockfile: נעילה יחידה ועדכנית; אין ghost deps.
	•	חיפוש Importים כפולים: Tailwind נטען פעם אחת; אין שני קבצי “כניסה” ל־CSS.
	•	בדיקת סטטיק: צד השרת מגיש סטטיק אחד; לא מגישים /auth גם מ־React וגם מ־Jinja.
	•	Lighthouse דגימה: להריץ אחת לכמה ימים על המסכים העיקריים—לאתר סחיבת ביצועים.
	•	ניטור עיצוב: לבדוק ב־Story/Playground שהטוקנים והקומפוננטים נשארו עקביים.

⸻

9) “הגדרת הצלחה” לפרויקט כולו (UI)
	•	אין אף כפילות בתצורה/טעינה (Tailwind/PostCSS/סטטיק).
	•	כל המסכים מצייתים לאותה שפה חזותית (טוקנים/Glass/רקע חי/Heebo/מיקרו־אינטראקציות).
	•	נגישות בסיסית מכוסה; מובייל־פרסט אמיתי; ביצועים מצוינים.
	•	“בנייה” של מסך חדש הופכת להיות עניין של שעות—not ימים—כי יש תבנית עיצובית יציבה וטוקנים מוגדרים.

⸻

איך למסור לבוט את המשימה (טמפלט קצר)
	•	“עדכן את כל התלויות ל־React 19, Vite 7, Tailwind 4.1; החלף את טעינת Tailwind ל־@import ב־CSS הראשי; העבר טוקנים ל־@theme; נקה פלאגינים/קבצים ישנים; ודא שאין Tailwind/PostCSS כפולים.
	•	הפוך את /auth ל־React בלבד (ללא Jinja); בדוק שאין סטטיק כפול.
	•	בנה מסך התחברות בשפה של ‘Glass+Gradient+Heebo+RTL+Motion’, עם ולידציה/נגישות/טוסטים וניתוב לפי role.
	•	הצג Evidence: צילום Lighthouse, צילום רספונסיביות, צילום נגישות (Tab flow), רשימת חבילות אחרי עדכון, ואישור שאין כפילויות בטעינת Tailwind/PostCSS/סטטיק.”

⸻
